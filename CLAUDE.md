
# CLAUDE.md — Next.js + SQLite SaaS

## Tech Stack

- **Runtime:** Node.js 22+ (Next.js 15 App Router)
- **Database:** better-sqlite3 via Drizzle ORM
- **Auth:** Lucia v3 (server-side sessions in SQLite)
- **UI:** Tailwind CSS v4 + shadcn/ui
- **Forms:** React Hook Form + Zod
- **Payments:** Stripe (checkout session pattern)

---

## ⚠️ Golden Rules

### 1. NEVER use `fetch()` for internal API calls
Use **Server Actions** (via `"use server"`) or direct function calls. Fetching your own API from the same server adds latency and bypasses type safety. The only exception is Stripe webhooks (must verify signature via raw body).

### 2. DB queries stay in `lib/db/queries/`
Each file exports one query group (e.g. `users.ts`, `teams.ts`, `subscriptions.ts`). Do NOT inline raw SQL or Drizzle calls inside Server Components or Actions — it breaks testability and reuse.

### 3. Every mutation must have a loading state
Server Actions return discriminated unions: `{ success: true, data: T } | { success: false, error: string }`. Client components call them in a `useActionState` wrapper. No bare `try/catch` swallows.

### 4. Migrations are immutable once merged
Once a Drizzle migration is applied to main, NEVER modify it. Create a new migration instead. This preserves audit trail and rollback paths.

## Naming Conventions

| What | Convention | Example |
|------|-----------|---------|
| Server Actions | `domain.action.ts` (kebab) | `billing.actions.ts` |
| DB queries | `entity.ts` (kebab) | `subscriptions.ts` |
| Components | PascalCase + kebab dir | `forms/PricingTable.tsx` |
| Tables | snake_case, plural | `team_members`, `subscription_items` |
| Columns | snake_case | `stripe_customer_id` |
| Types | PascalCase, `T` suffix | `UserT`, `TeamWithMembersT` |
| Env vars | `NEXT_PUBLIC_` prefix | `NEXT_PUBLIC_STRIPE_KEY` |

## Dev Commands

```bash
pnpm dev              # Local server on :3000
pnpm db:generate      # Generate Drizzle migration after schema change
pnpm db:migrate       # Apply pending migrations
pnpm db:studio        # Drizzle Studio GUI
pnpm lint             # Biome — no Prettier or ESLint
pnpm type-check       # tsc --noEmit
pnpm test             # Vitest (no Jest)
```

### DB Migration Workflow

1. Edit schema in `lib/db/schema/*.ts`
2. Run `pnpm db:generate` — creates timestamped migration
3. Review generated SQL in `lib/db/migrations/`
4. Run `pnpm db:migrate` locally
5. Commit both schema + migration together
6. Migrations auto-apply on deploy

### Anti-Patterns to Avoid

❌ `export default function Page()` — use named exports everywhere
❌ `"use client"` on the entire page tree — colocate client boundaries at leaf components
❌ `useEffect` for data fetching — use Server Components or `useActionState` + cache invalidation
❌ Raw SQL injection risks — always use Drizzle's parameterized queries
❌ Storing sessions in JWTs — Lucia sessions are server-side; use `lucia.validateSession()`
❌ `process.env` checks in the browser — prefix public vars with `NEXT_PUBLIC_`

### Patterns to Follow

✅ Server Components by default, `"use client"` only for interactivity
✅ Drizzle relations for JOINs — avoids raw JOIN strings
✅ Stripe Checkout Sessions — avoid PaymentIntents for subscription flows
✅ Shadcn toast for success/error feedback after mutations
✅ Rate-limit auth endpoints with `lucia.auth()` and `upstash-rate-limiter`
✅ Soft deletes via `deleted_at` timestamp — hard deletes only for test fixtures

## Why These Rules

- **No fetch() for internal API** — Server Actions preserve type safety and reduce latency by ~40% in App Router.
- **Drizzle over Prisma** — SQLite-native, faster cold starts, no binary dependencies on Lambda/Edge.
- **Lucia over Auth.js** — Full control over session table, no magic redirects, simpler SQLite integration.
- **Biome over ESLint** — 10x faster linting, zero config drift between workspaces.
- **pnpm** — Strict dependency isolation prevents phantom dependencies that break edge deployments.
